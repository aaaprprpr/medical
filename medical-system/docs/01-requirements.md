# 需求说明

## 1. 文档信息

| 项目 | 内容 |
| --- | --- |
| 项目名称 | 医学影像二分类系统原型 |
| 文档版本 | v0.1 |
| 文档状态 | 草案 |
| 适用阶段 | 本地开发与学习阶段 |

## 2. 项目背景

本项目基于已有医学影像二分类模型仓库，构建一个 Java Web 前后端分离系统原型。系统用于练习完整工程开发流程，包括前端页面、后端接口、数据库持久化、文件上传、跨服务调用和接口调试。

项目仅用于学习和原型验证，不用于真实医疗诊断。

## 3. 项目目标

- 提供一个简单的患者信息录入入口。
- 支持上传医学影像文件并发起模型推理。
- 调用本地 Python 模型服务，获得阳性或阴性二分类结果。
- 保存患者信息和检测记录。
- 支持查看患者列表、患者详情和历史检测结果。

## 4. 系统范围

### 4.1 本期范围

- 患者新增。
- 患者列表查询。
- 患者详情查询。
- 影像文件上传。
- 调用 Python 模型服务获取检测结果。
- 保存检测记录。
- 查看某个患者的检测记录。
- 本地前后端分离运行。

### 4.2 暂不支持

- 用户登录和权限管理。
- 医生账号体系。
- 正式医学报告生成。
- 云服务器部署。
- 多模型管理。
- 图像标注和复杂阅片功能。
- 医疗数据合规能力。

## 5. 用户角色

| 角色 | 说明 |
| --- | --- |
| 操作用户 | 本地使用系统录入患者、上传影像并查看检测结果的人 |
| 模型服务 | Python 推理服务，接收影像文件并返回二分类结果 |

## 6. 核心业务流程

1. 用户在前端录入患者姓名等基础信息。
2. 前端调用 Java 后端创建患者。
3. 用户选择某个患者并上传影像文件。
4. Java 后端接收文件并保存到本地目录。
5. Java 后端调用 Python 模型服务 `/predict`。
6. Python 模型服务返回二分类结果和置信度。
7. Java 后端保存检测记录。
8. 前端展示检测结果和历史记录。

## 7. 功能需求

### 7.1 患者管理

- 系统应支持创建患者。
- 系统应支持查询患者列表。
- 系统应支持查询单个患者详情。
- 患者至少包含 `id`、`name`、`createdAt`。
- 可选字段包含 `gender`、`age`。

### 7.2 检测管理

- 系统应支持为指定患者上传影像文件。
- 系统应支持调用模型服务并获取检测结果。
- 系统应保存影像路径、检测结果、置信度和检测时间。
- 系统应支持查询指定患者的检测历史。

### 7.3 模型服务调用

- Java 后端应通过 HTTP 调用 Python 模型服务。
- 模型服务地址应通过配置管理。
- 模型服务不可用时，后端应返回明确错误信息。

## 8. 非功能需求

- 本地开发环境应能独立运行前端、后端和模型服务。
- 后端接口返回格式应保持统一。
- 文件上传大小应可配置。
- 后端应保留必要日志，方便排查接口和模型调用问题。
- 第一阶段不追求高并发和复杂安全能力。

## 9. 验收标准

- 前端能成功调用 `GET /api/health`。
- 用户能创建患者并在列表中看到患者。
- 用户能为患者上传影像文件。
- 系统能调用假模型服务并返回阳性或阴性结果。
- 检测记录能被保存并再次查询。
- Java 后端、Vue 前端、Python 模型服务均可在本机运行。
