# 开发计划

## 1. 开发原则

- 先跑通最小闭环，再逐步补功能。
- 先用假数据和假模型服务，后接真实模型。
- 每完成一个接口，就用 Apifox 验证。
- 每个阶段都保持项目可启动、可测试。

## 2. 阶段计划

### M1：项目初始化

目标：

- 生成 Spring Boot 项目。
- 启动后端服务。
- 实现健康检查接口。

验收：

- `mvn spring-boot:run` 能启动。
- `GET /api/health` 返回 `status=UP`。

### M2：前端初始化

目标：

- 创建 Vue 项目。
- 配置前端本地端口 `5173`。
- 前端页面调用后端健康检查接口。

验收：

- 前端页面能展示后端健康状态。

### M3：患者管理

目标：

- 实现患者创建接口。
- 实现患者列表接口。
- 前端完成患者录入和列表展示。

验收：

- 能创建患者。
- 能在列表中看到创建后的患者。

### M4：数据库接入

目标：

- 接入 H2 或 MySQL。
- 建立 `patient` 表。
- 患者数据持久化。

验收：

- 后端重启后患者数据仍存在。

### M5：检测记录和文件上传

目标：

- 实现影像文件上传。
- 保存文件到本地 `uploads` 目录。
- 建立 `test_record` 表。
- 先用固定假结果保存检测记录。

验收：

- 上传文件后能生成检测记录。
- 能查询患者检测历史。

### M6：假模型服务

目标：

- 使用 Python FastAPI 实现 `/predict`。
- 接口随机返回 `POSITIVE` 或 `NEGATIVE`。
- Java 后端调用假模型服务。

验收：

- 上传文件后，检测结果来自 Python 服务。

### M7：真实模型接入

目标：

- 将假模型服务替换为真实医学影像二分类模型。
- 保持 `/predict` 接口契约不变。

验收：

- Java 后端无需修改接口逻辑即可调用真实模型。

### M8：前端完善

目标：

- 完善患者详情页。
- 展示检测结果、置信度、检测时间和图像预览。
- 增加简单错误提示。

验收：

- 用户能完整走通录入、上传、检测、查看历史流程。

## 3. 推荐开发顺序

1. `GET /api/health`
2. `POST /api/patients`
3. `GET /api/patients`
4. 前端患者录入和列表
5. 数据库持久化
6. `POST /api/patients/{patientId}/tests`
7. 假 Python `/predict`
8. Java 调 Python
9. 真实模型替换

## 4. 本地启动顺序

```text
1. 启动数据库
2. 启动 Python 模型服务
3. 启动 Spring Boot 后端
4. 启动 Vue 前端
```

开发早期没有数据库和模型服务时，可以只启动：

```text
1. Spring Boot 后端
2. Vue 前端
```
